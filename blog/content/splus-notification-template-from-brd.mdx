## Automation Name

**S+ Notification Template from BRD — Bilingual (AR/EN)**

This automation takes a client's S+ Configuration Manual or BRD (.pptx/.pdf/.docx) and cross-references it against the standard S+ notification template (.xlsx) to produce an enriched, client-specific notification file. Unlike the P+ approach (which removes unneeded rows), S+ templates are **additive** — the base template covers core events, and the BRD determines what additional events to add based on the client's strategy hierarchy, KPI levels, performance thresholds, and component structure.

## Step 1 — Cowork Extraction Prompt

Use this prompt in a **Claude Cowork** session (browser agent) to extract all notification events from a live S+ instance. Replace `{{URL}}` with the client's actual notifications page URL.

```text
Go to the following URL and extract ALL notification events for every component listed in the left sidebar:

{{URL}}

Follow these steps exactly:

1. Navigate to the URL and identify all components listed in the left sidebar
   (e.g., Objective L1, Objective L2, KPI L1, Initiative, Track, Project, Key Result, etc.).

2. For each component in the sidebar, click on it, then:
   a. Open the event dropdown (top-right "Select Event" combobox).
   b. Scroll through the entire dropdown to capture every single event option
      — do NOT assume events are the same across components.
   c. Record the exact event name as it appears in the dropdown.
   d. Check the existing template tab(s) below the dropdown — record the template
      name, channel, and any receivers already configured.
   e. Read the template body (the rich text editor content) and extract all template
      variables/properties used (anything in double curly braces like {{variableName}}).

3. Do NOT skip any component. Even if a component shows "No available options,"
   document that explicitly.

4. Use read_page with the listbox ref_id to get the full list of dropdown options
   when they are too long to fit on screen — visual screenshots alone may miss items.

5. Output Format — Compile everything into a single Markdown file saved to the
   outputs folder with this structure:

   ## For each component:
   - Component name and description
   - Table of ALL events (# | Event Name | Existing Template Name)
   - Template variables observed (variable name + description)

   ## At the end, include:
   - Summary of ALL unique event types grouped by category
   - Common template variables table (variable | description)
   - Notification channel used (Email, SMS, etc.)
   - Configured receivers (Owner, Champion, etc.)

6. Be thorough. Check every component, every event, every template.
   Do not assume components share the same events — verify each one individually.

Make sure to save the final output as a Markdown file in the outputs folder.
```

### URL Pattern Reference

```text
https://{client-subdomain}.masterteam.sa/template/{template-id}/notifications/{component-id}?eventId={event-id}&activeIndex={index}
```

| URL Part | Description | Example |
|----------|-------------|---------|
| `{client-subdomain}` | Client-specific subdomain | `g-smo`, `client-abc` |
| `{template-id}` | Strategy template number | `8`, `12` |
| `{component-id}` | Notification component ID | `122` = Objective L1 |
| `{event-id}` | Specific event ID (optional) | `1921` |
| `{active-index}` | Active template tab index (optional) | `0` |

## Step 2 — BRD Cross-Reference Prompt

This is the **core prompt**. Upload the S+ Notification Template and the client's BRD, then paste this. If you have Cowork extraction results from Step 1, add those as a third attachment.

```text
I have the following files:

1. An S+ Notification Template (.xlsx) — the standard bilingual notification template
   with 5 sheets: Objectives, KPIs, Initiatives & Tracks, Key Results, Variables Reference
2. A client BRD / Configuration Manual — describing the client's actual S+ strategy
   management system structure, components, hierarchy, and performance levels
3. (Optional) Cowork extraction results — a Markdown file with all notification events
   extracted from the client's live S+ system

Please cross-reference the BRD against the notification template and produce an
updated notification template that matches the client's actual system scope.

Requirements:
1. READ the BRD thoroughly — identify every strategy component, hierarchy level,
   KPI structure, performance threshold, and role that exists in the client's system
2. READ the notification template — understand every notification row across all sheets
3. CROSS-REFERENCE each BRD component against the template:
   - If the template already has the notification for a BRD component → KEEP as-is
   - If the BRD defines a component/event not in the template → ADD a new row
     following the exact same bilingual pattern and template variable conventions
   - Do NOT remove any existing rows from the template — only ADD when needed
4. PRESERVE the exact structure, formatting, column order, and bilingual content
   of the original template — do not restructure or restyle anything
5. PRESERVE all sheet names exactly as they are
6. If Cowork results are provided, validate that every event extracted from the live
   system has a corresponding notification row in the output template

Cross-reference checklist (map each BRD section to notification events):

OBJECTIVES SHEET — Check if the BRD has:
  - Objective Level 1 (created, updated, deleted)
  - Objective Level 2 (created, updated, deleted)
  - Objective Level 3 (created, updated, deleted)
  - OKR Objectives (created, updated)

KPIs SHEET — Check if the BRD has:
  - KPI Level 1 with Owner/Champion roles and frequency
  - KPI Level 2 with Owner/Champion roles and frequency
  - KPI Level 3 with Owner/Champion roles and frequency
  - Performance measurement (created, updated, measures updated, measure opened,
    measure value updated, daily lock reminder, one day before lock, locked, unlocked)
  - Performance thresholds / status changes (fully achieved, partially, not achieved)
  - KPI deletion events

INITIATIVES & TRACKS SHEET — Check if the BRD has:
  - Initiatives (added, updated, deleted)
  - Initiative Action Items (created, updated, deleted, progress updated)
  - Tracks (added, updated, deleted)
  - Track Action Items (created, updated, deleted, progress updated)
  - Projects (linked from P+ or standalone)

KEY RESULTS SHEET — Check if the BRD has:
  - Key Results (same measure lifecycle as KPIs)
  - Corrective Actions Plans (created, updated, deleted)
  - Performance status changes for Key Results

VARIABLES REFERENCE SHEET — Update with any new template variables introduced

After completing the cross-reference, list what was ADDED and why (with BRD slide/section
reference), then output the updated .xlsx file.
```

## Description

Every S+ (Strategy Management) system deployment requires a notification template that defines what email and notification centre alerts are sent for each strategic component lifecycle event. The standard template provides a comprehensive base covering Objectives (3 levels), KPIs (3 levels), Key Results, Initiatives, Tracks, and Projects — but each client's BRD may define additional components, performance thresholds, or roles that require new notification events.

Unlike P+ notifications (which are subtractive — you remove rows for features the client doesn't use), S+ notifications are **additive**. The base template covers core CRUD and measure lifecycle events. The BRD analysis adds client-specific events like performance status changes, deletion notifications, and corrective action plan events based on what the client's system actually defines.

### What Gets Generated

| Sheet | Base Template | After BRD Enrichment | Content |
|-------|--------------|---------------------|---------|
| **Objectives** | 8 rows | 8–11 rows | Objective create / update / delete for L1, L2, L3, OKR |
| **KPIs** | 27 rows | 27–33 rows | KPI lifecycle: create, update, measures, lock/unlock, status change, delete |
| **Initiatives & Tracks** | 13 rows | 13–15 rows | Initiative & Track CRUD + Action Item lifecycle |
| **Key Results** | 11 rows | 11–14 rows | Key Result lifecycle + Corrective Actions Plans |
| **Variables Reference** | 12 rows | 12–14 rows | All template variables with descriptions |

### Template Structure (All Sheets)

| Column | Content |
|--------|---------|
| A — Component | Component name (e.g., KPI L1, Objective L2, Initiative) |
| B — Event | Event trigger name from the system dropdown |
| C — Subject | Bilingual email subject (Arabic / English) |
| D — Email Body English | Full English email template with variables |
| E — Email Body Arabic | Full Arabic email template with variables |
| F — Notification Centre English | Short English in-app notification |
| G — Notification Centre Arabic | Short Arabic in-app notification |

### Notification Categories Mapped to BRD Sections

| Notification Category | What to Look For in BRD | Common BRD Section Names |
|----------------------|------------------------|-------------------------|
| Objectives L1/L2/L3 | Strategic hierarchy, objective levels, Admin/Business Admin roles | Objectives, System Structure |
| KPIs L1/L2/L3 | Performance indicators, Owner/Champion, frequency, polarity | KPIs, Performance Indicators |
| Key Results | Key results, OKR structure, corrective actions | Key Results |
| Performance Thresholds | Status ranges (>=99%, 85-99%, <85%) | Performance Levels |
| Initiatives | Initiative creation, Owner field, start/end dates | Initiatives |
| Tracks | Track structure linking initiatives to projects | Tracks |
| Projects | Projects linked from P+, contribution mapping | Projects |
| Action Items | Action item CRUD and progress tracking | Action Items |
| Corrective Actions | Corrective action plans for underperforming KRs | Corrective Actions |

### Key Benefits

- **59+ notification events audited automatically** — No manual BRD slide-by-slide comparison
- **Additive, not destructive** — Original template rows are never removed, only enriched
- **Bilingual integrity preserved** — Arabic and English content maintained in matching style
- **Performance threshold awareness** — Automatically adds status change notifications when the BRD defines performance levels
- **Cowork validation** — Optional live system extraction catches drift between BRD and actual system
- **Minutes not hours** — Complete cross-reference in a single Claude session (~2-3 minutes)
- **BRD-agnostic** — Works with any S+ client BRD regardless of format (.pptx, .pdf, .docx)
- **Audit trail** — Claude lists every added notification with the BRD section reference

## How to Use

### Option A — With Cowork Extraction (Recommended)

1. Open **Claude Cowork** (browser agent) and navigate to the client's S+ notifications page
2. Paste the **Step 1 Cowork Extraction Prompt** — replace `{{URL}}` with the actual URL

7. Review the addition summary and download the generated `.xlsx`

1. Open [Claude](https://claude.ai/new) (web or desktop app)
2. Attach **2 files**: notification template + client BRD
3. Paste the **Step 2 BRD Cross-Reference Prompt**
4. Review the addition summary and download the generated `.xlsx`
5. Open in Excel and verify the new rows match the BRD structure
6. Deliver to the configuration team

## Best Practices

- Always use the **latest version** of the client's BRD — if the BRD is outdated, notifications may be incomplete
- **Run Cowork extraction first** when possible — it catches events that exist in the system but may not be documented in the BRD
- **Do not edit the master template** — always start from a clean copy of `SPlus_Notifications_Template.xlsx`
- Pay special attention to the **performance levels section** — this determines whether status change notifications are needed
- Check if the client uses **OKR Objectives** separately from the L1/L2/L3 hierarchy
- For clients with **corrective action plans** on Key Results, ensure the full CRUD lifecycle is covered
- **Cross-reference Owner vs Champion roles** — some clients use both, some use only one

## Customization Options

Add any of these lines to the end of the prompt for specific adjustments:

- **Keep OKR rows:** `"The client uses OKR in addition to the 3-level objective hierarchy — keep all OKR notification rows"`
- **Add approval workflows:** `"The client has approval workflows for KPI measure submissions — add workflow notifications for approve/reject/return"`
- **Custom performance levels:** `"The client uses 4 performance levels instead of 3 — add status change notifications for each threshold crossing"`
- **Multiple BRDs:** `"I'm attaching 2 BRDs — one for the strategy hierarchy and one for KPI definitions. Cross-reference both"`
- **Include Cowork validation:** `"I'm also attaching the Cowork extraction results — validate every system event has a matching notification row"`

## S+ vs P+ — Key Differences

| Aspect | P+ Notification Template | S+ Notification Template |
|--------|------------------------|------------------------|
| **Approach** | Subtractive — remove unneeded rows | Additive — add BRD-specific rows |
| **Base template** | 98 rows covering all possible modules | 59 rows covering core strategy events |
| **Sheet structure** | 4 sheets: Level, Logs, Level WF, Log WF | 5 sheets: Objectives, KPIs, Init & Tracks, Key Results, Variables |
| **Hierarchy** | Portfolio > Program > Project | Objective L1 > L2 > L3 > KPIs > Key Results |
| **Workflows** | Approval cycles (add/update/delete/close) | Measure lifecycle (open > update > lock > unlock) |
| **Unique feature** | Phase Gate notifications | Performance threshold + Corrective Actions notifications |
